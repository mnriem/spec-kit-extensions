# Create Preset from Overrides

Package the project-local overrides in `.specify/templates/overrides/` into a shareable preset.

## Purpose

When developing a preset you often start by experimenting with project-local overrides — files placed in `.specify/templates/overrides/` that take highest priority in the template resolution stack. Once the overrides look good, this command turns them into a proper preset with a `preset.yml` manifest so they can be shared, version-controlled, and installed in other projects.

## User Input

$ARGUMENTS

The user may provide:
- A preset ID / name (e.g. `my-org-compliance`). If omitted, ask for one.
- An output directory. Defaults to the current working directory.
- Additional metadata (author, description). Prompted if not provided.

## Instructions

### 1. Discover Overrides

Scan the following locations for project-local overrides:

- **Template overrides**: `.specify/templates/overrides/` — any `.md` files here are template overrides.
- **Command overrides**: `.specify/commands/overrides/` — any `.md` files here are command overrides. If this directory does not exist or is empty, skip commands.

If **no override files are found** in either location, inform the user that there are no overrides to package and stop.

List all discovered files and show them to the user before proceeding.

### 2. Gather Preset Metadata

Collect the following from the user (use arguments if already provided, otherwise ask interactively):

| Field | Required | Default |
|-------|----------|---------|
| `id` | Yes | — |
| `name` | Yes | Derived from `id` (title-cased, hyphens to spaces) |
| `version` | No | `1.0.0` |
| `description` | Yes | — |
| `author` | Yes | — |
| `repository` | No | empty |
| `license` | No | `MIT` |
| `speckit_version` | No | `>=0.2.0` |
| `tags` | No | `[]` |
| `output directory` | No | `./<id>` in the current working directory |

The `id` must be lowercase, alphanumeric, and hyphen-separated only.

### 3. Create Preset Directory Structure

Create the output directory with the following structure:

```
<output-dir>/
├── preset.yml
├── README.md
├── templates/          # only if template overrides exist
│   └── <template files>
└── commands/           # only if command overrides exist
    └── <command files>
```

### 4. Copy Override Files

- Copy each file from `.specify/templates/overrides/` into `<output-dir>/templates/`.
- Copy each file from `.specify/commands/overrides/` into `<output-dir>/commands/` (if any exist).

Preserve the original filenames.

### 5. Generate `preset.yml`

Generate a valid `preset.yml` manifest. For each discovered file, create an entry under `provides.templates`:

- For **template** overrides: use `type: "template"`, set `name` and `replaces` to the filename without the `.md` extension.
- For **command** overrides: use `type: "command"`, set `name` and `replaces` to the filename without the `.md` extension.

Example generated manifest:

```yaml
schema_version: "1.0"

preset:
  id: "my-preset"
  name: "My Preset"
  version: "1.0.0"
  description: "Custom templates for our team workflow"
  author: "Jane Doe"
  repository: ""
  license: "MIT"

requires:
  speckit_version: ">=0.2.0"

provides:
  templates:
    - type: "template"
      name: "spec-template"
      file: "templates/spec-template.md"
      description: "Custom spec template"
      replaces: "spec-template"

    - type: "command"
      name: "speckit.specify"
      file: "commands/speckit.specify.md"
      description: "Custom specify command"
      replaces: "speckit.specify"

tags:
  - "custom"
```

### 6. Generate `README.md`

Generate a `README.md` in the output directory with:

- Preset name and description
- List of templates and commands included
- Installation instructions (`specify preset add --dev <path>`)
- A note that it was generated by Presetify and can be customized

### 7. Offer Cleanup

Ask the user whether to **remove the original override files** from `.specify/templates/overrides/` (and `.specify/commands/overrides/` if applicable). This avoids the overrides shadowing the preset once it is installed.

- If yes, delete the override files that were copied.
- If no, warn the user that the overrides will still take priority over the preset in the resolution stack and recommend removing them manually before installing the preset.

### 8. Show Next Steps

Tell the user:

1. Review and customize the generated `preset.yml` and `README.md`.
2. Test the preset locally:
   ```bash
   specify preset add --dev <output-dir>
   specify preset resolve spec-template
   ```
3. Once satisfied, version-control the preset and consider publishing it (see Spec Kit's [Preset Publishing Guide](https://github.com/github/spec-kit/blob/main/presets/PUBLISHING.md)).
